The official Anthropic repo. The canonical source for document processing (docx, pdf, pptx, xlsx) and many core creative and developer skills.

algorithmic-art

Generative art produced with code, in the spirit of p5.js, Processing, and generative design.

Use it when

  • You need a parametric event poster with 200 on-brand variations from one seed.
  • You want an SVG hero background that adapts to any aspect ratio via math instead of re-exporting a raster.
  • You are teaching or prototyping generative ideas and need a Voronoi tessellation or L-system without pulling in p5.js.

brand-guidelines

Applies Anthropic's official brand system (fonts, colors, spacing) to any artifact you generate.

Use it when

  • Drafting an internal Anthropic demo deck that needs exact brand colors, type, and spacing on every slide.
  • Rendering a one-pager artifact that must look official, with approved gradient, typography pairing, and accents.
  • Keeping weekly status updates visually consistent so the team never gets whiplash from rogue palette choices.

canvas-design

Produces PNG and PDF visual art using composition principles like balance, hierarchy, and typographic rhythm.

Use it when

  • Making a conference pull-quote graphic that balances a big quote, attribution, and negative space.
  • Generating a one-page memo cover PDF with proper hierarchy (kicker, title, subtitle, body) before it goes to print.
  • Composing a moodboard collage of five reference images with visual weight instead of a cluttered grid.

claude-api

Builds and tunes Anthropic SDK apps with prompt caching, tool use, and extended thinking.

Use it when

  • Scaffolding a new Claude-backed chatbot with prompt caching set up correctly on the first commit.
  • Migrating production code from Opus 4.6 to 4.7 and auditing tool schemas, caching breakpoints, and pricing deltas.
  • Diagnosing a 12% cache hit rate by tracing how your system prompt or tools invalidate on every request.

doc-coauthoring

Iterative collaborative drafting with awareness of version history and prior edits.

Use it when

  • Working through a 60-page product RFC across many sessions where earlier decisions must be respected.
  • Writing a technical book chapter over weeks with consistent voice, terminology, and back-references.
  • Merging feedback from three reviewers on a whitepaper without redrafting from scratch.

docx

Word document creation and editing with tracked changes, comments, and styles.

Use it when

  • Generating a legally formatted contract template with tracked changes on, numbered clauses, and heading styles.
  • Bulk-editing 50 consulting proposals where each needs the client name, date, and scope filled while preserving firm styles.
  • Extracting tracked changes and comments from a redlined doc so you can push a review summary to your PM tool.

frontend-design

UI patterns and implementation guidance, React and Tailwind heavy.

Use it when

  • Designing the information architecture and component tree for a new internal dashboard before opening Figma.
  • Recommending typography and spacing tokens for a landing page and emitting the Tailwind config that implements them.
  • Auditing an existing React library for missing states (loading, empty, error) and proposing the fixes.

internal-comms

Writes updates, newsletters, FAQs, and status reports in a consistent professional voice.

Use it when

  • Drafting a Monday "what I shipped" email in three styles so your manager picks the one that fits.
  • Writing a company-wide FAQ after a reorg where tone, what not to say, and pre-empted questions all matter.
  • Turning a chaotic Slack thread into a clean status update for the leadership digest.

mcp-builder

Scaffolds production-grade MCP servers in Python or TypeScript.

Use it when

  • Wrapping your internal billing API so Claude can query invoices and refunds with auth preserved.
  • Exposing a private Postgres reporting database as an MCP server with read-only guards, row limits, and timeouts.

pdf

Extracts tables, text, and metadata from PDFs, and can merge or annotate them.

obra/superpowers

Canonical 161K stars
View on GitHub

Jesse Obra's opinionated agentic software development methodology. Workflow discipline, not building blocks.

brainstorming

Socratic questioning that turns a vague idea into a concrete design.

Use it when

  • Starting a feature spec where you have a one-line idea ("user settings page") and need a scoped design in 20 minutes.
  • Opening move before a product pitch, where you need to articulate user, pain, alternative, and wedge.
  • Stuck in rabbit-hole thinking and wanting someone external to ask "what would cheaper look like" or "what if we shipped half."

dispatching-parallel-agents

Fans out independent work to many subagents in one shot.

Use it when

  • You have a PR ready and want security, perf, test-coverage, and docs reviews running at the same time.
  • Migrating a dependency across three microservices where each migration is independent but must complete before the lockfile bump.
  • Exploring three architecture options in parallel (REST vs GraphQL vs gRPC) and wanting each prototyped by a dedicated agent.

executing-plans

Runs multi-step plans with checkpoints and failure recovery.

Use it when

  • Running a 14-step production deploy runbook where step 3 must verify before step 4 fires.
  • Executing a three-phase database migration (add column, backfill, remove old column) with human pauses for verification.
  • Bringing up a new environment from IaC where any step's failure should halt and surface the cause, not continue blindly.

finishing-a-development-branch

Guided close-out flow: tests pass, review addressed, merge clean, branch deleted.

Use it when

  • You think the feature is done and want the pre-merge gauntlet: tests, lint, coverage, changelog, squash, PR body.
  • Handing a long-lived branch to another engineer so they don't inherit half-reverted files or stale lockfile entries.
  • Returning from a two-week vacation to a branch you started and can't remember the state of.

receiving-code-review

Framework for processing incoming feedback without flinching or getting defensive.

Use it when

  • Your PR has 40 comments from three reviewers and you need a triage: addressed, pushing back, rewrite, out of scope.
  • A senior reviewer left a terse "this is wrong" and you need to understand and respond without getting defensive.
  • You disagree with a reviewer and want to draft a technical rebuttal that stays collaborative.

requesting-code-review

Packages a diff with context, goals, and specific asks.

Use it when

  • Submitting a 2000-line refactor PR and wanting reviewers focused on the new service boundary, not adjacent test renames.
  • Asking a cryptography-focused colleague to specifically review the key-handling logic in your auth change.
  • Opening a draft PR for architectural feedback before investing in the implementation.

subagent-driven-development

Dispatches subagents per small task with code review gates between each.

Use it when

  • Building a CLI from scratch: arg parsing, config, command dispatch, output, each built and reviewed before the next.
  • Greenfielding a feature where drift between phases kills you; each subagent owns one slice with a review gate.
  • Porting a library to a new language where each module's port is a subagent with a correctness check against the original.

systematic-debugging

Methodical root-cause process: reproduce, isolate, hypothesize, test, confirm.

Use it when

  • A test fails on CI 1 in 20 runs and you need to rule out non-determinism, environment, and timing, one variable at a time.
  • Login works for 98% of users but 2% hit a 500; isolate the segment, reproduce, confirm root cause.
  • A customer says it's slow, your APM says it isn't, and the truth is in the middle.

test-driven-development

Strict red, green, refactor with enforcement of test-first order.

Use it when

  • Implementing a validation function where edge cases (empty, unicode, SQL injection) deserve tests before code.
  • Fixing a known bug: write the failing test that reproduces it, confirm red, fix to green, keep the test as a guard.
  • Refactoring a complex class: pin current behavior with characterization tests, then refactor with confidence.

using-git-worktrees

Creates isolated git worktrees with safe directory selection.

Use it when

  • Juggling three active PRs on the same repo where switching branches breaks your running dev server.
  • Running two Claude Code agents in parallel on the same repo without them stomping each other's files.
  • Keeping a pristine main worktree for quick git-bisect debugging while your feature worktree stays dirty.

sparse() {
  local repo="$1" target="$2" stage="$3"; shift 3
  local tmp; tmp=$(mktemp -d)
  git clone --filter=blob:none --sparse --depth 1 "$repo" "$tmp/r" >/dev/null 2>&1 && \
    (cd "$tmp/r" && git sparse-checkout set "$@" >/dev/null) && \
    cp -R "$tmp/r/$stage/." "$target/" && echo "  ok: $(basename "$repo")"
  rm -rf "$tmp"
}
cloneone() {
  local url="$1" target="$2"
  git clone --depth 1 "$url" "$target/$(basename "$url")" >/dev/null 2>&1 \
    && echo "  ok: $(basename "$url")" || echo "  skip: $(basename "$url")"
}
